Immutable

adjective - im·mu·ta·ble - \ (ˌ)i(m)-ˈmyü-tə-bəl \

not capable of or susceptible to change

Paradigm

Components are replaced rather than changed

Build new servers on every deploy and discard the old ones

Pets vs Cattle

“ In the old way of doing things, we treat our servers like pets, for example Bob the mail server. If Bob goes down, it’s all hands on deck. The CEO can’t get his email and it’s the end of the world. In the new way, servers are numbered, like cattle in a herd. For example, www001 to www100. When one server goes down, it’s taken out back, shot, and replaced on the line. ”
Randy Bias

The problem with traditional infrastructure

  • Code and runtime are separate
  • Configuration Drift
  • Manual changes and hacks leftovers
  • Running different versions of OS and packages
  • Entropy caused by many updates
  • Many points of failure
  • Failure events are costly and become fire drills
  • Services behave in unpredictable ways
  • Debugging the runtime is hard

How do we
solve this?

A new challenge

  • New tools are needed to build the system
  • High initial costs for automation
  • Architecture changes for applications

Application Architecture

12Factor APP

  • A methodology for building modern, scalable, maintainable software-as-a-service apps
  • General guidelines for building applications
  • Great ideas for working on the cloud

Config

  • Config is everything that is likely to change between deploys
  • Separate your config from your code
  • Store config in the environment
  • Use environment variables

Processes

  • Processes are stateless and share-nothing
  • Data that needs to persist must be stored in a stateful backing service
  • Filesystem of the process can be used as a brief, single-transaction cache

Disposability

  • Processes are disposable, meaning they can be started or stopped at a moment’s notice
  • Processes shut down gracefully when they receive a SIGTERM
  • Processes should also be robust against sudden death
