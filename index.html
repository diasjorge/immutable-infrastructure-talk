What is
immutable
infrastructure?

Immutable

adjective - im·mu·ta·ble - \ (ˌ)i(m)-ˈmyü-tə-bəl \

not capable of or susceptible to change

Paradigm

Components are replaced rather than changed

Build new servers on every deploy and discard the old ones

The problem with traditional infrastructure

Config

Config is everything that is likely
to change between deploys

  • Separate your config from your code
  • Store config in the environment
  • Use environment variables

Processes

Processes are stateless and share-nothing

  • Data that needs to persist must be stored in a stateful backing service
  • Filesystem of the process can be used as a brief, single-transaction cache

Disposability

  • Processes are disposable, meaning they can be started or stopped at a moment’s notice
  • Processes shut down gracefully when they receive a SIGTERM
  • Processes should also be robust against sudden death

Build, release, run

  • Strictly separate build and run stages
  • Build stage - Convert code repo into executable bundle
  • Release stage - Take the build and combine it with the configuration
  • Run stage - Execute the application launching processes

Creating builds

  • Bundle your code, dependencies and system packages into a single unit
  • We use packer + rpm/deb packages
  • Our builds use a hardened version of amazon linux and ubuntu

Releasing Code

  • Loading configuration
  • Retrieve config from storage system
  • We use AWS dynamodb + KMS
  • We use Strongbox for managing secrets

Creating a cluster

Node lifecycle

  • All nodes boot from same image
  • Loads configuration from environment and key/value store
  • Generate configuration files for processes that need them.
  • Gathers information about the cluster from cloud provider
  • If it’s the first node bootstraps the cluster
  • Joins an existing cluster as a new node or another node’s replacement
  • Respond to healthcheck to indicate success

Rolling Updates

  • Small framework extensible for many cluster types
  • Currently supports Cassandra, Kafka and Zookeeper

Update cloud
infrastructure

  • Deploy infrastructure stack
  • Create resources, permissions, network rules

Updating instances

  • Check cluster health
  • Functional tests + monitoring information
  • Kills and instance gracefully
  • Stop receiving connections
  • Wait for new node to be ready
  • Wait for the cluster recovery and proceed with next node

Embrace your cloud provider

  • We use many AWS services
  • Cloudformation for infrastructure as code
  • API to gather cluster information
  • SSM for sending commands
  • Dynamodb for configuration
  • S3 for storing backups

Summary

Advantages

  • Known state and less deployment failures
  • Testing infrastructure in isolation is possible
  • Consistent environments between production and staging / development
  • Easy to rollback

Trade offs

  • High upfront investment
  • Rolling update of a whole cluster takes time

Take aways

  • Immutable infrastructure is a means to an end not a goal.
  • It’s not about the tools it’s about the ideas.
  • You can implement it gradually.
  • It may not be a good fit for you.
